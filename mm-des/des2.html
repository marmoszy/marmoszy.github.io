mm-des - minimalistic discrete event simulator

MM February 2024 (v.1), September 2025 (v.2)

The implementation of a discrete event simulator with a BPMN-like elements and Python-like scripting. The simulator uses text-defined elements (events, tasks, gates) and their connections as an input for execution. The results are in the form of animated or static BPMN diagram, the timeline for each element and the text table with the time of process execution along with final values of scenario variables.

mm-des allows to run simulations of process models defined in text form as the list of elements and the list of connections. The code implements eight basic BPMN elements and six random generators:

  1. Start(fun, param, tnmax, script) - start event
  2. Task(fun, param, script, nservers=1) - sequential task activity with n-servers inside
  3. XorGate(script) - exclusive gateway (two-output: value==true passes the event to the first defined connection, value==false - to the second (more than two: must have algebraic choice-like function with result being a number starting from 0) multi-input: pass trough the event from all connected), default script is "=B(0.5)" meaning binomial equally distributed choice for two outputs
  4. AndGate(script) - parallel gateway (multi-output: passes through the event to all connected, multi-input: waits for all)
  5. Timer(fun, param, script) - timer delay event
  6. Condition(script) - conditional event with a script containing logical expression as the first one ("=<expr>;..."); the instructions after first semicolon will be executed only when condition evaluates to true
  7. Terminate(script) - terminate event
  8. End(script) - end event
where:
  • fun - random functions: E([mean]), N([mean,std]), U([min,max]), T([min,max,mean,std]), B(p), C([p1,p2,p3,...]), None
  • param - array of fun params (fun==None: deterministic time delay in param[0])
  • tnmax - tnmax>0: maximum time for start generator; tnmax<0: -tnmax represents maximum number of generated events
  • script - string with semicolon separated codes for evaluation during event execution "[var]=<expr>; ... ;", empty var defaults to 'value' variable which is tested in XorGate and Condtion, var starting with 'S.' are scenario ones, starting with 'A.' element ones and without prefix - token/customer ones. Variable 'A.n' is element event counter which is initiated with 0 and increased by 1 at exec of each token/customer event. Token/customer have variable 'cname' (customer name) and may have variables '__t<id>a', '__t<id>b' and '__t<id>e' when the token visited the element with identifier id. They mark arrival, begin and end times.
The text description of a process contains list of elements that should be ordered by number (element identifier) and semicolon separated list of connections. The elements should be numbered using left to right rule. It allows for using simple automatic layout algorithm for presentation (which is not perfect). In case of presence of overlaying elements in final diagram it is possible to force position of an element by adding slash-separated '/x[/y]' values after event identifier. Moreover, one-line description of connections could be split into multi-line one and then the first appearance of element number in the connection forces the y position of the element to be according to line sequence. This approach is recommended for more complicated processes as it helps to achieve nice-looking diagram.

Minimum working example (MWE) with default values for all events looks like: 

# A single task
1 Start()
2 Task()
3 End()
1->2; 2->3
Default parameters for Start() are (E,[1],50) what generates exponential events with mean of 1 time unit ongoing 50 time units. Default values for Task() are (U,[1,2]) what means that this task takes from 1 to 2 time units uniformly distributed. As a result the bpmn-like schematic is generated along with single process implementation diagram and the time of process execution: 

#n t
1 76.31941273749922

When defining a process it is possible to use '#' as the comment character. If the first line starts with the # it plays a role of title of the process. When using '#' after definition of an element it plays a role of the comment that could be added to the bpmn-like diagram above the element. The next '#' in the same line allows to add additional comment that is not visible in the diagram. See next section with examples.

The section Running code allows for defining a process and running simulation. The examples presented in next section could be inserted by clicking numbered buttons. Before running simulation it is possible to define the number of simulations. When only one simulation is selected, animated bpmn diagram is generated along with timeline chart. It allows for detailed analysis of token flows in the process. When the number of simulations are greater than 1 the bpmn execution timeline is not generated. It is possible to configure the bpmn-like diagram to include token animations, comments and the first script command.
